# Philosophy Evolve

基于充分的案例证据和模式分析，提议对 `PHILOSOPHY.md` 进行结构化演进。这是整个反思系统中门槛最高的操作——必须有统计证据支撑，且需人工确认方可执行。

## 核心约束

- **任何 PHILOSOPHY.md 变更必须经人工确认**，无例外
- 变更以**模块级重写或新增独立模块**为单位，禁止碎片化追加单句
- 变更必须有**充分证据链**：≥ 3 个案例 + 明确因果 + strong 模式评级
- 被拒绝的提议标记为 `rejected`，不可就同一模式重复提议（除非有新增案例改变了判断）

---

## 触发场景

| 类型 | 条件 |
|------|------|
| **模式驱动** | pattern-detect 检测到 strong 级别模式，其 pattern 文件中包含 philosophy 变更建议 |
| **用户主动** | 用户明确要求修改投资哲学 |

**不触发**：
- 单笔亏损反思后直接提议修改（必须走 case → pattern → evolve 完整链路）
- weak/moderate 模式（证据不足）

---

## 执行流程

### Step 1: 整理证据链

组装完整的变更论据：

| 证据项 | 来源 | 必填 |
|--------|------|------|
| 触发模式 | `/app/custom/reflection-workspace/lessons/patterns/{pattern_id}.md` | 是 |
| 关联案例 | `/app/custom/reflection-workspace/lessons/cases/{case_id}.md`（全部关联案例） | 是 |
| 当前哲学条目 | `/app/custom/research-workspace/PHILOSOPHY.md` 中将被变更的章节原文 | 是 |
| 对核心池的影响 | 变更后可能影响的核心池标的数量和方向 | 建议 |

### Step 2: 确定变更方式

哲学变更**仅允许**以下两种方式：

| 方式 | 适用场景 | 说明 |
|------|---------|------|
| **模块重写** | 已有章节覆盖该主题但规则不够具体/需修正 | 保留章节位置和标题层级，重写内容 |
| **新增模块** | 现有章节未涵盖的新维度 | 新增独立小节，有明确边界和适用条件 |

**禁止的变更方式**：
- 在已有段落末尾追加一两句话
- 插入零散注释或脚注
- 创建"杂项"或"其他"类兜底模块

### Step 3: 草拟变更方案

变更方案需包含：

```
## Philosophy 变更提议

### 变更依据
- **触发模式**：{pattern_id} — {模式描述}
- **案例数量**：{n} 个（{case_id 列表}）
- **时间跨度**：{first_seen} ~ {last_seen}
- **累计影响**：亏损 {总金额} 元 / {总比例}%

### 变更目标
**文件**：/app/custom/research-workspace/PHILOSOPHY.md
**目标章节**：§{章节号} {章节名}
**变更方式**：模块重写 / 新增模块

### 变更前（当前内容）
{原文引用}

### 变更后（建议内容）
{完整的新内容，保持与 PHILOSOPHY.md 整体风格一致}

### Diff 摘要
- 新增了什么：...
- 修改了什么：...
- 删除了什么：...

### 影响评估
- **对研究流程的影响**：{这一变更如何改变未来的研究行为}
- **对核心池的影响**：{是否导致部分标的需要重新评估}
- **对交易审查的影响**：{trade-plan-review 的审查标准是否需要相应调整}
- **逆向风险**：{这一变更是否可能导致过度保守或过度约束}
```

### Step 4: 对话中展示方案

将完整的变更提议在对话中展示，明确等待用户确认：

```
---
⚠️ **Philosophy 变更提议**

{变更方案全文}

---
**请确认是否执行此变更。**
回复"确认"将写入 PHILOSOPHY.md；回复"拒绝"将记录为已拒绝。
```

### Step 5: 执行或记录

#### 用户确认 → 执行写入

1. 按变更方案修改 `/app/custom/research-workspace/PHILOSOPHY.md`
2. 更新 pattern 文件的 `status` 为 `confirmed`
3. 在对话中确认写入完成
4. 提醒用户：是否需要据此重新评估核心池中的相关标的

#### 用户拒绝 → 记录

1. 更新 pattern 文件的 `status` 为 `rejected`
2. 记录拒绝原因（如用户提供）
3. 在对话中确认已记录
4. **同一模式不再重复提议**，除非：
   - 新增 ≥ 2 个同类案例（累积证据显著增强）
   - 用户主动重新提起

## 质量要求

### 变更内容的风格一致性

写入 PHILOSOPHY.md 的新内容必须与现有文档风格保持一致：
- 使用相同的标题层级和编号体系
- 使用相同的表格格式
- 保持相同的语言风格（精炼、观点鲜明、有判断力）
- 维度和框架保持同层次抽象程度

### 避免过度反应

- 不因单一类型的亏损就全面收紧标准
- 新增规则必须明确**适用边界**，防止误杀
- 评估逆向风险：过于保守可能导致错过机会，同样是一种损失

### 变更的可回溯性

每次变更在 pattern 文件中留有完整记录（变更前后内容 + 决策理由），确保未来可以回溯和审视历史演进路径。

## 与其他技能的协同

- **pattern-detect** → 提供 strong 模式和变更建议草案
- **loss-reflect** → 提供底层案例证据

- research-workspace 的 **research-quality-scoring** → 评分标准可能因哲学变更而调整
- trading-workspace 的 **trade-plan-review** → 审查维度可能因哲学变更而调整
