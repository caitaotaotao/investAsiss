# Search for Research

投资研究领域的深度搜索与多源信息收集技能。采用迭代式深度搜索策略，从宏观、中观、微观三个层次系统性收集投研所需信息，为个股卡片和行业卡片提供高质量数据输入。

## 核心约束

> **⚠️ 搜索引擎硬性规则（必须遵守，无例外）**
>
> 1. **通用搜索必须用 Bocha**：所有不含 `site:` 的搜索查询，**必须使用 `bocha` 子命令**，禁止使用 `search` 子命令
>    ```bash
>    # ✅ 正确：通用查询用 bocha
>    python /app/custom/scripts/sogou_search.py bocha "公司名 关键词" --count 15
>    # ❌ 错误：通用查询用 search（禁止）
>    python /app/custom/scripts/sogou_search.py search "公司名 关键词"
>    ```
> 2. **仅 `site:` 查询用搜狗**：包含 `site:` 或 `-site:` 的查询才使用 `search` 子命令
>    ```bash
>    # ✅ 正确：site: 查询用 search
>    python /app/custom/scripts/sogou_search.py search "site:mp.weixin.qq.com 公司名 纪要"
>    ```
> 3. **脚本路径**：始终使用 `/app/custom/scripts/sogou_search.py`（非 research-workspace 路径）

**必须先读** `PHILOSOPHY.md` — 搜索方向和信息筛选必须围绕投资哲学框架，避免收集与投资逻辑无关的噪音信息。

## 设计理念

参考业界高星开源项目的核心架构思想：

- **迭代式深度搜索**：借鉴 dzhng/deep-research 的 depth/breadth 递归搜索模式，每轮搜索基于上一轮结果动态调整方向
- **任务分解与自校验**：借鉴 virattt/dexter 的 task planning + self-validation 模式，将复杂研究问题拆解为结构化子任务，每步校验信息质量
- **多源数据工具协同**：借鉴 schnetzlerjoe/hermes 的多 agent 多工具协作模式，从财报、宏观、行情、新闻等多维度交叉验证

## 搜索参数配置

每次深度搜索任务启动前，根据研究目标确定参数：

| 参数 | 说明 | 推荐值 | 适用场景 |
|------|------|--------|---------|
| **breadth（广度）** | 每轮并发搜索的查询数量 | 3-6 | 广度越大覆盖面越全，但耗时越长 |
| **depth（深度）** | 递归搜索的最大层数 | 2-4 | 深度越大挖掘越深，适合复杂主题 |
| **focus（聚焦维度）** | 搜索聚焦的分析层次 | macro/sector/company | 决定搜索的起始方向 |

### 预设搜索模板

| 模板名 | breadth | depth | focus | 用途 |
|--------|---------|-------|-------|------|
| **快速扫描** | 3 | 1 | 单一维度 | 快速了解某个话题或验证单一信息点 |
| **标准调研** | 4 | 2 | company 或 sector | 为建档提供基础信息 |
| **深度研究** | 6 | 3 | 全维度 | 重大投资决策前的全面调研 |
| **事件追踪** | 3 | 3 | 事件链 | 追踪某一事件的前因后果和影响链 |

## 搜索工具：双引擎检索脚本

本技能使用 `/app/custom/scripts/sogou_search.py` 作为核心搜索工具，支持**搜狗搜索**和 **Bocha WebSearch API** 双引擎。

- **搜狗搜索（`search`）**：基于爬虫，免费，**唯一支持 `site:` 语法**的引擎，适合精准定位特定信源
- **Bocha WebSearch API（`bocha`）**：付费 API，**结果质量显著高于搜狗**、结构化更好、带时间戳，是**通用搜索的首选引擎**

### 搜索引擎选择策略：场景驱动双引擎

不再是"搜狗优先、Bocha 降级"，而是**根据查询类型选择最优引擎**：

```
查询是否包含 site: 语法？
├── 是 → 使用搜狗（Bocha 不支持 site: 语法）
│         └── 搜狗结果为空或不足？→ 去掉 site: 后用 Bocha 重试
└── 否（通用搜索）→ 使用 Bocha（质量更高、带时间戳）
                      └── Bocha 异常（API 报错/超时）？→ 降级用搜狗
```

| 场景 | 首选引擎 | 原因 |
|------|---------|------|
| **通用关键词搜索** | **Bocha** | 结果质量高、结构化好、带时间戳 |
| **宽泛探测（阶段零）** | **Bocha** | 探测需要高质量结果来发现新线索 |
| **时效性敏感查询**（最新业绩、近期事件） | **Bocha** | 结果带 `date` 字段，可判断信息新旧 |
| `site:mp.weixin.qq.com` 站内搜索 | 搜狗 | Bocha 不支持 `site:` |
| `site:xueqiu.com` 等站内搜索 | 搜狗 | 同上 |
| 排除黑名单 `-site:` 搜索 | 搜狗 | Bocha 不支持 `-site:` |
| 搜狗 `site:` 搜索结果为空 | **Bocha** | 去掉 `site:`，改用 `{关键词} {信源名}` 做宽泛搜索 |

### AI 搜索操作流程

执行搜索任务时，按以下步骤使用脚本：

1. **判断引擎** — 根据查询类型选择搜索引擎：
   - 查询**不含** `site:` / `-site:` → **使用 Bocha**（通用搜索首选）
   - 查询**包含** `site:` / `-site:` → 使用搜狗（Bocha 不支持 site 语法）

2. **执行搜索（Bocha 优先）** — 通用关键词搜索**直接用 Bocha**，无需先试搜狗：

   ```bash
   # 通用搜索 → 直接 Bocha（结果质量高、带时间戳）
   python /app/custom/scripts/sogou_search.py bocha "贵州茅台 2025年报 业绩" --count 15

   # 站内搜索 → 搜狗（Bocha 不支持 site: 语法）
   python /app/custom/scripts/sogou_search.py search "site:mp.weixin.qq.com 贵州茅台 路演纪要"
   ```

3. **评估搜索结果质量** — 检查返回结果数量和相关性：
   - Bocha 结果充分且相关 → 继续步骤 4
   - Bocha 异常（API 报错/超时/0 结果） → **降级用搜狗重试**：
     ```bash
     python /app/custom/scripts/sogou_search.py search "贵州茅台 2025年报 业绩" --pages 2
     ```
   - 搜狗 `site:` 搜索结果为空 → **去掉 site:，用 Bocha 做宽泛搜索**：
     ```bash
     python /app/custom/scripts/sogou_search.py bocha "宁德时代 深度分析 雪球" --count 10
     ```

4. **阅读结果列表** — 浏览返回的标题、来源域名和摘要，筛选出符合信息源白名单的高质量结果

5. **深入阅读** — 对有价值的链接使用 `fetch` 命令获取全文内容

   ```bash
   python /app/custom/scripts/sogou_search.py fetch "https://xueqiu.com/xxx/article"
   ```

6. **站内搜索** — 配合 `site:` 语法精准定位白名单信息源（仅搜狗支持）

   ```bash
   python /app/custom/scripts/sogou_search.py search "site:xueqiu.com 宁德时代 深度分析"
   python /app/custom/scripts/sogou_search.py search "site:mp.weixin.qq.com 半导体 产业链"
   ```

7. **排除黑名单** — 搜索时主动排除低质量源（仅搜狗支持 `-site:` 语法）

   ```bash
   python /app/custom/scripts/sogou_search.py search "比亚迪 销量 -site:guba.eastmoney.com -site:baijiahao.baidu.com"
   ```

### 注意事项

- **Bocha 是通用搜索首选**：结果质量显著高于搜狗、结构化好、带 `date` 字段（爬取时间）便于判断信息时效性。**所有不含 `site:` 的通用查询必须首选 Bocha**
- **搜狗仅用于 site: 场景**：Bocha 不支持 `site:` / `-site:` 语法，站内搜索和排除黑名单搜索使用搜狗
- **请求频率**：搜狗脚本内置随机延迟（1-2.5 秒/页），避免触发反爬。单次搜索建议不超过 5 页
- **来源校验**：结果中的 `source` 字段即为域名，用于快速判断是否命中黑名单
- **URL 解析**：搜狗默认会解析重定向链接为真实 URL；加 `--no-resolve` 可跳过以提升速度
- **正文提取**：`fetch` 自动去除导航、广告等非正文内容，默认最多返回 15000 字符
- **降级策略**：Bocha 异常（API 报错/超时）时降级用搜狗；搜狗 `site:` 搜索 0 结果时改用 Bocha 宽泛搜索

## 搜索工作流程

### 阶段零：宽泛探测（Open Exploration）— 个股研究强制执行

> **核心原则（归纳法优先）**：研究一只股票时，**先假设你不知道它是什么**，用宽泛搜索探测所有可能的业务方向，再基于探测结果决定研究框架。绝不能先入为主地给公司贴标签（如"油气装备龙头"），然后只在标签范围内搜索。
>
> **反面案例**：杰瑞股份（002353）首次建档时，预设"油气装备"标签，所有搜索词围绕油气设计，彻底遗漏了北美 AIDC 现场发电需求带来的燃气轮机第二增长曲线。根因：用了"演绎法"（从已知推未知），而 AIDC 是全新增量信息，需要用"归纳法"（从信息推结论）。

**适用条件**：任何个股**首次研究/建档**时，本阶段为**强制执行**，不可跳过。

#### Step 1：探测搜索（3 组必做）

| 组别 | 关键词模板 | 目的 |
|------|----------|------|
| **新业务/转型探测** | `{公司名} 新业务`、`{公司名} 转型`、`{公司名} 第二增长曲线`、`{公司名} 新赛道 {year}` | 发现公司是否在进入传统认知之外的新领域 |
| **市场热点交叉** | `{公司名} AI`、`{公司名} AIDC`、`{公司名} 数据中心`、`{公司名} 新能源`、`{公司名} 出海`、`{公司名} 机器人`、`{公司名} 低空经济` | 用当前市场热点词与公司名交叉探测未知关联 |
| **一手信源探测** | `site:mp.weixin.qq.com {公司名} 路演纪要 {year}`、`{公司名} 互动易 新业务`、`{公司名} 投资者关系活动记录 {year}` | 从管理层/分析师一手信息中发现研报尚未覆盖的新方向 |

> **"市场热点交叉"动态更新**：上述热点词并非固定列表，应根据当前 A 股市场主流投资主题动态增补（如 2025 年末可加"人形机器人"、"固态电池"等）。

#### Step 2：公司官网 + 交易所互动易（强制检查）

探测阶段**必须**主动检查以下一手信源，搜索引擎索引存在系统性盲区：

| 信源 | 检查方法 | 价值 |
|------|---------|------|
| **最新业绩公告/定期报告** | 在巨潮资讯网（cninfo.com.cn）搜索公司最新年报/半年报/季报 PDF，用 `fetch` 获取或搜索 `{公司名} {year} 年报 管理层讨论与分析` | **年报/半年报中"管理层经营讨论与分析"章节是最核心的一手信息**——管理层会在该章节系统阐述各业务线的经营现状、新业务进展、战略布局、风险因素，信息密度远高于研报摘要 |
| **公司官网新闻** | `site:{公司官网域名} 订单 OR 签约 OR 新产品` 或直接 `fetch` 官网新闻页 | 公司自行发布的订单/合作/新产品，搜索引擎可能未收录 |
| **交易所互动易** | `site:irm.cninfo.com.cn {公司名}` 或 `{公司名} 互动易 投资者提问` | 投资者提问常涉及未被广泛关注的新业务、订单、产能信息 |
| **投资者关系活动记录** | `{公司名} 投资者关系活动记录表 {year}` | 机构调研内容摘要，常最早披露新业务进展 |

> **为什么必须读"管理层经营讨论与分析"？** 年报/半年报的该章节由管理层亲自撰写，是上市公司对全部业务（含新业务）最完整、最权威的一手表述。研报是分析师的二手解读，可能存在选择性忽略；而"管理层经营讨论与分析"会逐一披露各业务线的收入、成本、产能、在建项目、战略方向。**杰瑞股份案例中，AIDC 燃气轮机业务在年报该章节已有明确表述，但因首次建档时未阅读年报原文而被完全遗漏。**

#### Step 3：探测结果处理

| 结果 | 处理 |
|------|------|
| **发现新业务线索** | 记录线索，纳入阶段一子问题清单，作为**独立研究维度**深入。新线索可能比传统主业更重要（如杰瑞 AIDC） |
| **未发现新线索** | 正常进入阶段一，在卡片中标注"已执行宽泛探测，未发现已知业务之外的新方向" |
| **搜索返回 0 结果** | **绝不等于"无此事"**，必须执行零结果自修正 |
| **仅返回传统业务描述** | 不满足探测目的，视同未完成——换用更具体的热点词或一手信源继续探测 |

#### 零结果自修正规则

> **"0 结果"是强负面信号**：意味着搜索词方向可能错误、信息源覆盖不足，**而非"该事物不存在"**。每次遇到 0 结果，必须反思："是我的搜索词有问题？还是信息源有盲区？"

| 触发条件 | 修正动作 |
|---------|---------|
| 搜狗搜索 0 结果 | 切换 Bocha API 重新搜索同一主题 |
| 关键词组合 0 结果 | 拆分关键词、换用同义词/行业术语重试（如"燃气轮机"→"gas turbine"→"分布式发电"→"现场发电"） |
| `site:` 限定搜索 0 结果 | 去掉 `site:` 限制，用通用搜索补充 |
| 中文搜索 0 结果 | 补充英文搜索（尤其公司有海外业务/客户时） |
| 连续 2 次换词仍 0 结果 | 判定该方向暂无公开信息，标注"信息盲区"并记录，继续下一方向 |

> 此零结果自修正规则同样适用于后续阶段一、阶段二中的所有搜索步骤。

### 阶段一：问题分解（Task Decomposition）

将用户的研究问题拆解为结构化子问题，明确搜索意图：

1. **识别研究目标类型**
   - 个股研究 → 对接 `stock-rationale-card` 的信息需求
   - 行业研究 → 对接 `sector-rationale-card` 的信息需求
   - 主题研究 → 围绕特定投资主题（如"国产替代"、"AI算力"）
   - 事件研究 → 追踪某一事件对投资逻辑的影响

2. **生成子问题清单**

   以个股研究为例，自动拆解为：

   ```
   [核心问题] 公司XXX是否值得纳入核心池？
   ├── [子问题0] ★ 阶段零探测发现的新业务/新增长曲线（如有）的具体规模与前景？
   ├── [子问题1] 公司主营业务和核心竞争力是什么？（含阶段零发现的新业务）
   ├── [子问题2] 成长驱动因素（量/价/效率）分别是什么？（按全部业务线拆分）
   ├── [子问题3] 所处行业的周期位置在哪里？
   ├── [子问题4] 管理层的战略能力和执行力如何？
   ├── [子问题5] 近期有哪些关键催化剂或风险事件？
   └── [子问题6] 当前估值水平是否合理？
   ```

   > **子问题0 来自阶段零**：如果阶段零探测到了新业务线索，必须生成对应的子问题并优先研究。新业务可能改变整个投资逻辑的定价锚。

3. **为每个子问题生成初始搜索查询**
   - 每个子问题生成 2-3 个不同角度的搜索查询
   - 查询语言：优先中文（A股标的），辅以英文（全球视野）
   - 查询策略：混合使用精确关键词和自然语言问题

### 阶段二：迭代搜索（Iterative Deep Search）

采用广度优先 + 深度递归的搜索策略：

```
第 1 层（广度搜索）
├── 查询1 → 结果 → 提取关键发现 + 新方向
├── 查询2 → 结果 → 提取关键发现 + 新方向
├── 查询3 → 结果 → 提取关键发现 + 新方向
└── 汇总本层发现 → 生成下一层查询

第 2 层（深度递归）
├── 基于第1层发现的新查询1 → 更深入的结果
├── 基于第1层发现的新查询2 → 更深入的结果
└── 汇总 → 判断是否需要继续深入

第 N 层（达到 depth 上限或信息饱和时停止）
└── 最终汇总所有发现
```

**每轮搜索的处理步骤**：

1. **执行搜索** — 通用查询**直接使用 `bocha` 命令**（结果质量更高）；仅含 `site:` 的查询使用 `search` 命令。必要时用 `fetch` 深入阅读
2. **零结果检查** — 若返回 0 结果，**不得跳过该方向**，按"阶段零 · 零结果自修正规则"换词/换引擎/换语言重试
3. **提取关键发现（Learnings）** — 从搜索结果中提炼与投资逻辑相关的关键信息
4. **识别新方向（Directions）** — 发现需要进一步探索的线索，尤其关注是否出现**与事前认知不一致的信号**（如发现未知的新业务、新客户、新应用场景）
5. **信息质量评估** — 判断信息来源可靠性、时效性、与投资逻辑的相关性
6. **认知偏差自检** — 本轮搜索是否只在已有认知框架内搜索？是否需要跳出框架做发散性搜索？
7. **决策：继续/停止** — 信息是否饱和？是否还有关键缺口？

**搜索终止条件**：
- 达到 depth 上限
- 信息饱和（连续两轮无实质性新发现）
- 子问题已全部覆盖且信息充分
- **阶段零探测已完成**（个股研究时，必须确认宽泛探测已执行，不能在未探测的情况下终止）

### 阶段三：信息整合与校验（Synthesis & Validation）

1. **去重与合并** — 去除重复信息，合并同一主题的多源发现
2. **交叉验证** — 对关键数据点进行多源交叉确认（符合 PHILOSOPHY.md 3.1 多重交叉验证原则）
3. **标注置信度** — 对每条关键发现标注信息质量等级
4. **识别信息缺口** — 明确哪些关键问题尚未获得充分信息
5. **生成结构化输出** — 按标准模板输出搜索结果
6. **🗜️ 上下文压缩（独立搜索任务时）** — 若本次搜索为独立任务（非 stock-rationale-card/sector-rationale-card 子流程），搜索结果输出后按 AGENTS.md「Session 上下文管理协议」触发 context compaction。若作为卡片建档的子流程运行，由卡片技能在最终步骤统一触发压缩

## 搜索结果输出规范

搜索结果**不单独持久化存储**，而是直接在对话中以结构化格式呈现，供用户判断后再由下游技能（`stock-rationale-card`、`sector-rationale-card`）吸收到对应的 knowledge 文件中。

### 输出格式

搜索完成后，在对话中按以下结构输出：

**1. 搜索概述**
- 研究问题、搜索范围（宏观/中观/微观）、实际搜索轮次
- 信息完整度评估（高/中/低）及缺口说明

**2. 核心发现**（按重要性排序）
- 每条发现标注置信度（高/中/低）、信息来源及时间
- 与 PHILOSOPHY.md 的关联：周期位置判断、成长驱动验证、风险因素更新

**3. 详细发现**
- 按宏观/行业/公司分层组织

**4. 信息质量评估**
- 关键信息点的来源等级（P0-P4）、时效性、是否交叉验证

**5. 信息缺口与后续建议**
- 未覆盖的关键问题、建议的后续行动

## 特殊搜索场景

### 事件驱动搜索

当突发事件（政策变化、行业事故、公司公告等）发生时：

1. **事件本身** — 搜索事件的完整信息和官方表态
2. **历史对标** — 搜索类似历史事件及其后续影响
3. **影响链条** — 沿产业链搜索上下游受影响环节
4. **市场反应** — 搜索市场即时反应和机构观点
5. **持续跟踪** — 标记需要持续关注的后续节点

### 财报季搜索

财报发布期间的高效搜索策略：

1. **管理层经营讨论与分析（年报/半年报必读）** — 获取并阅读年报/半年报原文中"管理层经营讨论与分析"章节，逐业务线理解经营现状、新业务进展、产能/订单变化、战略布局及风险因素。这是信息密度最高的一手来源
2. **业绩速读** — 搜索收入/利润/现金流核心指标 vs 市场预期
3. **边际变化** — 搜索环比/同比的关键变化及管理层解释
4. **指引更新** — 搜索管理层对未来季度的业绩指引
5. **同行对比** — 搜索同行业已披露财报的共性趋势
6. **预期差定位** — 结合搜索结果判断实际业绩与市场预期的差距

### 主题投资搜索

围绕特定投资主题（如"AI应用落地"、"出海"）的搜索：

1. **主题定义** — 明确主题边界和核心逻辑
2. **产业链映射** — 搜索主题涉及的完整产业链图谱
3. **受益排序** — 搜索不同环节的受益程度和确定性
4. **催化剂日历** — 搜索未来可能兑现的关键节点
5. **风险评估** — 搜索主题可能被证伪的条件

## 质量要求

- **归纳法优先**：个股首次研究时，必须先完成阶段零宽泛探测再进入结构化搜索，禁止跳过探测直接按"已知标签"搜索
- **零结果必修正**：搜索返回 0 结果时，禁止直接放弃该方向，必须换词/换引擎/换语言重试
- **认知偏差自检**：每轮搜索后自问"本轮是否只在已有认知框架内搜索？是否需要跳出框架？"
- **信息时效性**：优先使用近 3 个月内的信息，超过 6 个月的数据需标注"历史数据"
- **来源标注**：每条关键信息必须标注来源和日期，无来源信息不得作为决策依据
- **交叉验证**：关键数据点至少需要 2 个独立来源确认
- **投资相关性**：所有搜索必须服务于 PHILOSOPHY.md 框架下的投资逻辑判断，避免收集无关信息
- **增量价值**：每轮搜索应产生增量信息，避免重复搜索已知内容
- **偏见识别**：标注信息来源的潜在立场偏差（如卖方研报的利益冲突、媒体的流量导向）
- **一手信源覆盖**：公司官网和交易所互动易是搜索引擎的盲区，个股研究时必须主动检查
